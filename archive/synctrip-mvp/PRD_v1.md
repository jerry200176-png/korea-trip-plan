產品需求文件 (PRD) - 旅遊協作 App v1.0 (MVP)
一、 產品願景與定位
一句話描述： 專為雙人或小型群組打造的「即時地圖與行程協作白板」。

要解決的核心問題： 消除跨平台收集資訊的碎片化，以及通訊軟體上來回討論行程的溝通摩擦。

二、 核心使用者故事 (User Story)
情境範例： 一個平日忙於處理補習班業務與系統架設的工程師，下班後在永和的家中打開 App，希望能與人在大安區的女友詠絮即時連線。兩人可以在同一個畫面上，搜尋 Google 地圖上的餐廳，並拖曳到週末的時間軸裡。不需要在 LINE 上互相丟連結、截圖確認，就能直覺、無縫地排定出遊計畫。

三、 MVP 核心功能範圍 (In-Scope)
為確保第一版能快速上線測試水溫，v1.0 僅包含以下三大模組：

1. 基礎帳號與行程管理 (Trip Management)
功能描述： 讓使用者能建立一趟旅程，並邀請他人加入。

User Flow：

支援免密碼快速登入（例如 Google OAuth 或 Apple Sign-in，降低註冊門檻）。

建立新行程：輸入「旅程名稱」、「開始/結束日期」。

協作邀請：產生一組專屬邀請連結或邀請碼，另一人點擊後即可加入該行程成為共同編輯者。

1. 地圖整合與景點沙盒 (Map & Sandbox)
功能描述： 將找景點與排行程的動作分開，提供一個暫存靈感的地方。

User Flow：

畫面分為上半部「地圖視圖」與下半部「操作面板」。

使用者可在 App 內透過關鍵字搜尋地標（串接 Google Places API）。

點擊地標後，可選擇「加入待定沙盒（Wishlist）」。加入後，地圖上會多出一個大頭針。

1. 即時同步的時間軸排程 (Real-time Timeline)
功能描述： 類似 Google Docs 的體驗，兩人同時編輯時畫面會即時更新。

User Flow：

提供以「天」為單位的垂直時間軸。

使用者可將「待定沙盒」中的景點卡片，長按並拖曳（Drag and Drop）到時間軸上的特定時段。

輸入或修改「預計停留時間」時，透過 WebSocket 廣播，另一人的畫面需在 1 秒內同步更新位置與資料。

四、 非 v1.0 範圍：殘酷的「減法清單」(Out of Scope)
身為系統開發者，一定會手癢想加功能，但以下這些在 MVP 階段絕對不能做，以控管開發成本：

❌ IG/小紅書連結自動解析爬蟲： 技術複雜度太高，且有被鎖 IP 的風險。初期請使用者乖乖用內建搜尋找地標。

❌ 多幣別公費拆帳系統： 牽涉到金錢計算的關聯資料庫設計極度複雜，容易產生 Bug 導致客訴。初期請使用者先用其他記帳 App。

❌ 自動計算交通時間與路線防呆： 呼叫 Google Maps Distance Matrix API 成本較高，且路徑最佳化演算法耗時。初期只要能呈現「景點 A 接著景點 B」順序即可，交通時間由使用者自己抓。

❌ 投票與遊戲化機制： 屬於錦上添花的功能，不影響「把行程排出來」的基礎價值。

五、 技術架構需求提示 (Technical Requirements)
資料庫： PostgreSQL（需支援 PostGIS 以便未來擴充地理空間查詢）。

即時通訊： 建立 WebSocket 伺服器處理連線狀態。

狀態管理： 由於涉及即時拖曳，前端需有強大的狀態管理機制（如 Redux、Zustand 或 Bloc），並處理「樂觀 UI 更新（Optimistic UI Updates）」——即拖曳當下畫面先變動，背景再打 API 寫入資料庫，若失敗再回滾，以確保操作滑順。
